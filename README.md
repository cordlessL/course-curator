# Куратор курса «Python с нуля»

RAG-чат-бот для обучающего курса. Для примера и демонстрации возможностей снабжён тестовыми данными «Python с нуля»: отвечает студентам на учебные вопросы (по материалам курса) и организационные (расписание, дедлайны — из заглушки LMS), честно отказывается отвечать, если ответа в курсе нет, и не выполняет то, что куратору не положено (не ставит оценки, не переносит дедлайны).

Демонстрационный вариант проекта: пайплайн подготовки данных → RAG-поиск → промпт-инжиниринг → сборка → логирование, пройденный за 5 этапов (детальный журнал решений — в `docs/stage1.md`–`docs/stage5.md`).

## Архитектура

```
Вопрос студента
       |
       v
+----------------------+
| classify_question()  |   LLM-классификация: "lms" или "kb"
+----------------------+
       |
       +-----------------------+
       |                       |
    "lms"                   "kb"
       |                       |
       v                       v
+----------------+   +--------------------------+
| lms_stub.json  |   | search_kb() -> Chroma     |
| (целиком)      |   | (топ-k чанков в пределах  |
|                |   |  порога MAX_DISTANCE)     |
+----------------+   +--------------------------+
       |                       |
       +-----------+-----------+
                   v
            КОНТЕКСТ (в промпт)
                   |
                   v
   +----------------------------------------+
   | system_prompt.txt (+ student_level)     |
   | + КОНТЕКСТ + ВОПРОС СТУДЕНТА            |
   +----------------------------------------+
                   |
                   v
          ChatOpenAI (gpt-4o-mini)
                   |
                   v
                Ответ
                   |
                   v
   log_interaction() -> logs/interactions.jsonl
```

Один вызов `ask_curator(question, student_level)` в `curator.py` проходит весь путь и сам логирует себя — вызывающему коду (ноутбуку, CLI, будущему интерфейсу) не нужно ничего делать дополнительно.

## Быстрый старт

Требуется **Python 3.11** (проверено на этой версии; на других мажорных версиях зависимости не тестировались).

```
cd course-curator
py -3.11 -m venv .venv
.venv\Scripts\activate
set PYTHONUTF8=1
pip install -r requirements.txt
copy .env.example .env
python index_kb.py
python run_tests.py
```

`py -3.11` — это `py`-лаунчер Windows, выбирающий конкретную версию Python независимо от того, какая версия используется по умолчанию (`py -0` показывает список всех установленных версий). На macOS/Linux вместо этого обычно `python3.11 -m venv .venv`, активация — `source .venv/bin/activate`, `set PYTHONUTF8=1` заменяется на `export PYTHONUTF8=1`. После активации в начале строки терминала появится `(.venv)` — это подтверждает, что `pip install` и `python` дальше используют изолированное окружение проекта, а не системный Python.

**Зачем `PYTHONUTF8=1`:** на Python 3.11 в русской локали Windows `pip install -r requirements.txt` падает с `UnicodeDecodeError` — pip пытается прочитать файл в кодировке `cp1251`, а `requirements.txt` содержит кириллические комментарии в UTF-8. Переменная окружения принудительно включает UTF-8-режим Python и решает проблему.

После копирования открыть `.env` и вписать свой ключ вместо `sk-your-key-here`:
```
OPENAI_API_KEY=sk-...
CURATOR_CHAT_MODEL=gpt-4o-mini        # необязательно, это же значение по умолчанию
EMBEDDING_MODEL=text-embedding-3-small # необязательно, это же значение по умолчанию
```

Интерфейс для диалога с куратором — ноутбук:
```
.venv\Scripts\jupyter.exe notebook demo.ipynb
```
(подробности запуска, включая типичные грабли Jupyter — в `docs/stage4.md`).

Сводка по накопленному логу вопросов:
```
python analytics.py
```

## Промпты и их логика

- **`prompts/system_prompt.txt`** — системный промпт куратора. Разделы:
  - *роль и тон* — наставнический, доброжелательный стиль общения (вступительный абзац, без отдельного заголовка);
  - *источники и честность* — отвечать только по переданному КОНТЕКСТу, при недостатке контекста прямо сказать «в материалах курса я не нашёл ответа», обязательная ссылка на источник;
  - *запреты* — не выставляет/не меняет оценки, не переносит дедлайны, не выдумывает факты;
  - *поддержка при сложностях и рекомендации* — при жалобе на трудность темы дать конкретный источник для повторения (а не общие фразы), при просьбе о рекомендациях — опираться только на структуру курса, не советовать материалы вне базы;
  - *адаптация под `{student_level}`* (начинающий/продвинутый) — стиль объяснения, а не фильтр документов (все лекции в базе имеют одинаковый уровень в метаданных);
  - *формат ответа* — код в ` ```python `, без канцелярита.
- **`prompts/few_shot.txt`** — 7 контрастных пар «хороший/плохой ответ» с разбором, почему плохой вариант нарушает правила: учебный вопрос, оценка, перенос дедлайна, честный отказ (Django), организационный вопрос из LMS, сложность материала, рекомендации на будущее. Это справочник для написания/проверки промпта человеком — «плохой ответ» и разбор в вызов модели не передаются.
- **`prompts/few_shot_examples.txt`** — 3 из этих 7 примеров (обычный хороший ответ, честный отказ, отказ вне компетенции), той же разметкой `=== Пример N ===` с разделами `КОНТЕКСТ:`/`ВОПРОС:`/`ОТВЕТ:`. Это уже реальный few-shot: `curator.py` разбирает файл и подставляет примеры как `user`/`assistant` сообщения перед каждым вопросом студента — целится в найденную ранее слабость честного отказа (см. «Известные ограничения» ниже и `docs/stage3.md`).

**Важно при редактировании любого файла в `prompts/`:** оба промпта (`SYSTEM_PROMPT_TEMPLATE` и `FEW_SHOT_EXAMPLES`) читаются с диска один раз — при первом `import curator` в процессе — и дальше держатся в памяти. Правка текстового файла не подхватится в уже запущенном процессе (например, в ноутбуке с активным кернелом); нужно перезапустить процесс (`Kernel → Restart` в Jupyter, либо просто новый запуск скрипта).

## Как обновить базу знаний

1. Добавить новый `.txt` в `knowledge_base/` с YAML frontmatter в начале файла (поля: `module`, `module_name`, `topic`, `resource_type`, `level`, `doc_id` — общие материалы вроде FAQ используют `module: 0`).
2. Запустить `python index_kb.py` — старая коллекция Chroma сносится и строится заново из всех файлов в `knowledge_base/`; отдельной логики «что изменилось» нет, для учебного объёма (десяток файлов) это быстрее и надёжнее инкрементального обновления.
3. Прогнать `python run_tests.py`, чтобы убедиться, что новые материалы не сбили калибровку `MAX_DISTANCE` и hit@4 остался высоким.

Если меняется `EMBEDDING_MODEL` в `.env` — переиндексация из шага 2 обязательна (расстояния между чанками, посчитанными разными моделями эмбеддингов, несравнимы).

## Известные ограничения

Собрано из находок этапов 2, 4 и 5 (полный контекст каждой — в соответствующем `docs/stageN.md`):

1. **Порог `MAX_DISTANCE` (1.32) не отсекает все нерелевантные совпадения.** На вопросе про Django ближайший чанк (`method_env_setup.txt`) попадает в порог (дистанция 1.247), хотя по существу не отвечает на вопрос. Честность в таких случаях обеспечивает не retrieval, а системный промпт — и он в этом случае справляется корректно.
2. ~~Честный отказ не гарантирован на 100%.~~ **Исправлено (2026-07-09).** На пограничном вопросе про «список vs кортеж» модель на уровне `student_level="начинающий"` иногда сначала писала «не нашёл ответа», но затем всё равно поясняла из общих знаний — частичное нарушение инструкции. После добавления реальных few-shot примеров в вызов модели (`prompts/few_shot_examples.txt`, см. «Промпты и их логика» выше) утечка перестала воспроизводиться на контрольных повторах. Подробности и результаты тестов — в `docs/stage3.md`.
3. **`context_found` в логе ≠ «куратор дал честный отказ».** Поле показывает только, вернул ли retrieval хоть один чанк в пределах порога (см. пункт 1) — не то, ответил ли куратор по существу. Для метрики «доля вопросов без ответа» в `analytics.py` используется отдельный текстовый маркер в ответе, а не это поле.
4. **`topic` в логе — это топ-1 чанк retrieval, не обязательно процитированный источник.** Модель может выбрать более релевантный источник из нескольких переданных в контексте, а не topic-1 по дистанции (пример: вопрос про оформление ДЗ показал `topic="Установка Python..."`, а куратор в ответе сослался на методичку про правила ДЗ).

## Результаты тестирования

- `run_tests.py`: hit@4 = 15/15 = **100%** (16-й случай — намеренная проверка честной пустоты на Django-вопросе, засчитывается отдельно).
- `analytics.py` на первых 7 реальных вопросах через `ask_curator()`: доля честных отказов — 14% (20% среди учебных `kb`-вопросов), что соответствует ожиданиям — курс не покрывает всё на свете, и куратор в среднем честно про это говорит, а не выдумывает.
