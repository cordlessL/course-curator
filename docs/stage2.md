# Этап 2 — RAG-пайплайн (индексация и поиск)

## Что сделано

Три скрипта + `requirements.txt`:

- **`index_kb.py`** — читает txt, отрезает YAML frontmatter регуляркой, валидирует типы метаданных (Chroma принимает только str/int/float/bool), режет `RecursiveCharacterTextSplitter` (размер чанка 800, перекрытие 150). В сепараторы первым добавлен `"\n== "` (маркер заголовков секций в лекциях) — режет по смысловым границам. Переиндексация = повторный запуск (старая коллекция сносится и строится заново).
- **`search_kb.py`** — функция `search_kb(query, k, module, resource_type)`, собирает `where`-фильтр Chroma (`$and` при нескольких условиях), отсекает чанки за порогом расстояния `MAX_DISTANCE`. Пустой результат — осознанный сигнал «ответа в базе нет». Модуль сделан импортируемым — на этапе 4 `ask_curator()` делает `from search_kb import search_kb`.
- **`run_tests.py`** — 16 контрольных случаев: 13 обычных вопросов (покрывают все файлы, включая переформулированные), 2 вопроса с фильтрами по метаданным, 1 вопрос без ответа в базе (проверка честной пустоты). Метрика — hit@4, цель ≥85%.

Секреты — через `.env` + `python-dotenv` (`load_dotenv()` в `index_kb.py` и `search_kb.py`). `.gitignore`: `.env`, `chroma_db/`, `__pycache__/`, `.venv/`.

## Результат

- Индексация: 36 чанков из 8 txt-файлов, без ошибок.
- Калибровка `MAX_DISTANCE`: поднят с 1.3 до **1.32** — минимум, при котором проходит валидный чанк про срезы списков (дистанция 1.308).
- Итоговый hit@4 по 15 содержательным вопросам — **100%**.

## Принятые решения и почему

- **Порог `MAX_DISTANCE` = 1.32, а не туже** — потому что ложное срабатывание на вопросе про Django (дистанция 1.270) оказалось *ниже* дистанции валидного чанка про срезы (1.308). Случаи пересекаются в пространстве эмбеддингов — одним порогом расстояния это не разрулить.
- **Архитектурное решение:** честность («в базе ответа нет») не форсируется на уровне retrieval, а перекладывается на системный промпт куратора (этап 3). Retrieval — это грубый фильтр, а не единственная линия защиты от галлюцинаций.
- **`.venv` изначально не пересоздавали** после переименования папки проекта (`curse-curator` → `course-curator`), так как переименование сломало консольные лаунчеры (`pip.exe`) внутри старого окружения — тогда обошлись через `python -m pip install -r requirements.txt` без пересоздания.
- **Проверено отдельно (2026-07-08):** `.venv` удалён и пересоздан с нуля уже в переименованной папке (`python -m venv .venv` + `.venv/Scripts/python.exe -m pip install -r requirements.txt`) — установка прошла чисто, лаунчеры не ломаются. Переиндексация и `run_tests.py` повторно дали 36 чанков из 8 файлов и hit@4 = 15/15 = 100% (единственный «провал» — ожидаемый Django-кейс). Значит поломка лаунчеров была следствием именно переименования уже существующего `.venv`, а не проблемой самого окружения или структуры проекта.

## Порядок локального запуска

```
pip install -r requirements.txt
python index_kb.py
python run_tests.py
```

`OPENAI_API_KEY` задаётся не через переменную окружения вручную, а в файле `.env` (читается через `python-dotenv`, `load_dotenv()` вызывается в `index_kb.py` и `search_kb.py`).

## Обновление (после этапа 4): модель эмбеддингов вынесена в `.env`

`EMBEDDING_MODEL` в обоих файлах теперь читается как `os.getenv("EMBEDDING_MODEL", "text-embedding-3-small")` — значение по умолчанию оставлено как fallback-константа в коде, чтобы всё продолжало работать и без переменной. Само значение задаётся один раз в `.env` (`EMBEDDING_MODEL=text-embedding-3-small`) и общее для `index_kb.py` и `search_kb.py` — это принципиально, так как модель эмбеддингов на поиске обязана совпадать с моделью на индексации, иначе расстояния между чанками теряют смысл.

**Важно:** смена значения `EMBEDDING_MODEL` требует полной переиндексации (`python index_kb.py`) — иначе часть или все чанки в Chroma останутся с эмбеддингами от старой модели. Пользователь сознательно оставил этот шаг за собой и выполнил его вручную (2026-07-08) — переиндексация прошла успешно, база в порядке.

## Обновление (2026-07-09): `.venv` закреплён на Python 3.11

По требованию пользователя `.venv` пересоздан на конкретной версии `py -3.11 -m venv .venv` (было — Python 3.14.5, версия, случайно взятая по умолчанию на этапе 2). Причина фиксации версии не задокументирована пользователем явно, но согласуется с тем, что в других его проектах (`pcef9-main`) тоже используется `.venv` на Python 3.11 — вероятно, предпочтение по совместимости версии по умолчанию для проектов.

**Находка:** на Python 3.11 в русской локали Windows `pip install -r requirements.txt` падает с `UnicodeDecodeError: 'charmap' codec can't decode byte... character maps to <undefined>` — pip пытается прочитать `requirements.txt` (содержит кириллические комментарии в UTF-8) в кодировке `cp1251` вместо UTF-8. На Python 3.14 эта проблема не проявлялась. Решение — выставить `PYTHONUTF8=1` перед `pip install` (в README добавлено `set PYTHONUTF8=1` в шаги быстрого старта). Переустановка всех зависимостей и повторный `run_tests.py` после пересоздания `.venv` — hit@4 = 15/15 = 100%, без регрессий.

## Обновление (2026-07-09): фильтр поиска по теме (`topic`)

При сверке с исходными функциональными требованиями обнаружилось, что `build_filter()`/`search_kb()` поддерживали фильтрацию только по `module` и `resource_type`, хотя метаданные каждого чанка содержат ещё и `topic` — фильтр по теме отсутствовал. Добавлен параметр `topic` в обе функции (точное совпадение со значением поля `topic` из YAML-шапки файла, не текстовый поиск) и флаг `--topic` в CLI. Проверено вручную: `python search_kb.py "как добавить элемент в список" --topic "Списки и словари, индексация, срезы, методы"` — фильтр корректно сузил выдачу до `lec_m4_collections.txt`. `run_tests.py` после изменения — без регрессий (hit@4 = 100%).
