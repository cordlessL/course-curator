# Этап 5 — Логирование и аналитика

## Что сделано

- **`logger.py`** — функция `log_interaction(...)`, дописывает одну строку в `logs/interactions.jsonl` на каждый вызов `ask_curator()`. Поля записи: `timestamp` (UTC, ISO), `question`, `student_level`, `question_type` (`lms`/`kb`), `topic`, `sources` (список файлов-источников), `context_found` (bool), `answer`.
- **`curator.py`** обновлён: `build_kb_context()` теперь возвращает не только текст контекста, но и `(sources, topic)` — нужны только для лога, не для самого промпта. `ask_curator()` в конце вызывает `log_interaction()`.
- **`logs/`** добавлена в `.gitignore` (генерируемые данные, та же логика, что и `chroma_db/`).
- **`analytics.py`** — сводка по логу без pandas (`Counter`): топ тем по частоте, доля честных отказов (по тексту ответа, не по `context_found`). Подробности ниже.

## Принятые решения и почему

- **Формат — JSONL, а не CSV.** Поле `sources` — список переменной длины (0, 1 или несколько источников на вопрос); в JSON он ложится нативно, в CSV потребовал бы либо отдельных колонок source_1/source_2/…, либо сериализации списка строкой внутри ячейки. Дозапись в JSONL — это просто добавление строки в конец файла (`open(path, "a")`), без перечитывания и переписывания всего файла, как потребовалось бы для одного JSON-массива.
- **`ask_curator()` логирует сама, без отдельного вызывающего кода.** Логирование — не опциональный шаг, а часть контракта функции: любой вызов (из ноутбука, CLI, будущего интерфейса) автоматически попадает в лог без дополнительных действий со стороны вызывающего кода.
- **`logs/` не кладём в git** — как и `chroma_db/`, это генерируемые данные, а не код или конфигурация проекта.

## Важная находка: `context_found` — не то же самое, что «честный отказ»

При тестовом прогоне Django-вопроса (`"Как развернуть Django-приложение на сервере?"`) лог показал `context_found=True`, хотя куратор корректно ответил честным отказом («в материалах курса я не нашёл ответа»).

Причина — это тот же пограничный случай из этапа 2: `search_kb()` всё-таки нашёл один чанк (`method_env_setup.txt`, дистанция 1.247) в пределах порога `MAX_DISTANCE=1.32`, то есть контекст технически не пуст. Честность ответа в этом случае обеспечивает не retrieval, а системный промпт куратора (этап 3) — модель сама решила, что найденный контекст по существу не отвечает на вопрос.

**Следствие:** `context_found` — это сигнал «retrieval вернул хоть что-то», а не «куратор ответил по существу вопроса». Для метрики из пайплайна «доля вопросов без ответа в базе» (этап 6 плана) поля `context_found` недостаточно — нужен отдельный сигнал, основанный на тексте самого ответа (например, наличие фразы-маркера honest-refusal, как в системном промпте: «я не нашёл ответа на этот вопрос»), а не только на факте непустого контекста.

**Как учесть дальше:** при построении сводной аналитики (следующий шаг этапа 5) считать «долю вопросов без ответа» по признаку текста ответа, а не по `context_found`. Само поле `context_found` в логе оставить как есть — оно полезно отдельно (для отладки retrieval), просто не должно использоваться как прямая замена признаку отказа.


## Аналитика: `analytics.py`

Простая сводка на основе `logs/interactions.jsonl`, без pandas (не нужна лишняя зависимость ради 2-3 показателей — используется `collections.Counter`).

- `load_log()` — читает все записи JSONL.
- `topic_counts()` — топ тем по частоте вопросов (LMS-вопросы считаются под темой `"LMS"`, а не по теме конкретного модуля).
- `is_refusal(answer)` / `refusal_rate()` — доля честных отказов, определяется по текстовому маркеру в ответе (`"не нашёл/нашла/нашли ответа"` — формулировка из `system_prompt.txt`), а НЕ по `context_found` (см. находку выше в этом файле про Django-случай).
- `print_summary()` — печатает сводку: всего вопросов, топ тем, доля отказов (общая и отдельно среди `kb`-вопросов).

Запуск: `python analytics.py`.

### Находка: `topic` в логе может не совпадать с реально использованным источником

На вопросе «Как правильно оформить ДЗ?» топ-1 чанк по дистанции retrieval оказался из `method_env_setup.txt` (тема «Установка Python и настройка VS Code»), и именно эта тема попала в лог. Но в самом ответе куратор корректно сослался на другой, более релевантный источник — `method_homework_rules.txt` («Правила оформления и сдачи домашних заданий») — то есть LLM сама выбрала более уместный чанк из нескольких переданных в контексте, а не обязательно топ-1 по расстоянию.

**Следствие:** поле `topic` в логе отражает ранжирование retrieval (топ-1 результат), а не обязательно источник, на который куратор реально опёрся в ответе. Для аналитики топ тем это приемлемая приближённая метрика (тема запроса примерно понятна), но не стоит использовать `topic` как точный аудиторский след того, что процитировано в ответе — для этого пришлось бы парсить строку «Источник: …» из самого текста ответа, что не реализовано (усложнение сверх задачи простой сводки этапа 5).

## Обновление (2026-07-10): расширены маркеры честного отказа в `is_refusal()`

При подготовке чистого примера отчёта для документации (после архивации шумного тестового лога и прогона 12 реалистичных вопросов) поймали живой пример именно той эвристической слабости, что была отмечена как находка выше: на вопрос про рекурсию (тема не покрыта курсом) куратор честно отказался — «В материалах курса я не нашёл **информации** о рекурсии...» — но `is_refusal()` ловил только формулировку «не нашёл **ответ**» (как в `system_prompt.txt`), поэтому посчитал это не отказом. `refusal_rate()` на свежем логе из 12 вопросов показывал 0% вместо честных ~8%.

**Решение:** `REFUSAL_MARKERS` расширен вариантами «не нашёл/нашла/нашли» + `информаци-`/`данны-` (основы без окончаний, тот же приём, что уже был с «ответ» → ловит «ответа»/«ответ на» через `in`). Список — не исчерпывающее решение (LLM может перефразировать и дальше), а расширение по факту увиденного плюс правдоподобные варианты.

Проверено: пересчёт на том же логе из 12 вопросов — 8% (1/12) и 10% среди kb-вопросов (1/10), рекурсия теперь учитывается верно. Прогон новых маркеров на архивном логе из 38 записей (`logs/interactions_testing_2026-07.jsonl.bak`) — ложных срабатываний не найдено.
