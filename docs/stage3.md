# Этап 3 — Архитектура промптов

## Что сделано

Папка `prompts/` с двумя файлами (тексты промптов не зашиты в код Python):

- **`prompts/system_prompt.txt`** — системный промпт куратора курса «Python с нуля» (реальное название берётся из `lms_stub.json`, `course.title`). Разделы:
  - источники и честность (отвечать только по переданному КОНТЕКСТу, честно говорить «не нашёл ответа», обязательная ссылка на источник — модуль/тип материала или "LMS");
  - запреты (не ставит/не меняет оценки, не переносит дедлайны, не выдумывает факты);
  - адаптация под `{student_level}` (начинающий/продвинутый — переименовано из «уверенный» 2026-07-09, см. обновление ниже);
  - формат ответа (код в ```python, без воды).
- **`prompts/few_shot.txt`** — 5 контрастных пар «хороший/плохой ответ» с пояснением, почему плохой вариант нарушает правила:
  1. обычный учебный вопрос с ответом в базе;
  2. вопрос про оценку/балл;
  3. просьба продлить дедлайн;
  4. вопроса нет в базе (Django-кейс — прямое применение вывода с этапа 2 про честный отказ);
  5. организационный вопрос с ответом из LMS-заглушки.

  **Важно:** `few_shot.txt` — это справочный материал для написания/проверки системного промпта (и заготовка для документации на этапе 6), а не диалоговая история для передачи в API. На этапе 4 его не нужно подставлять в вызов LLM как есть — только использовать как ориентир при формулировке промпта.

## Принятые решения и почему

- **`student_level` передаётся параметром вызывающего кода, а не определяется моделью самостоятельно** — подставляется в системный промпт через плейсхолдер `{student_level}`. Все лекции в базе имеют одинаковый `level: начинающий` в метаданных, так что адаптация — это стиль генерации, а не фильтр документов.
- **Тексты промптов — отдельные файлы в `prompts/`**, как и было заложено в структуре проекта на этапе 1 — чтобы промпты можно было редактировать без правки Python-кода.
- **Классификация вопроса (учебный/организационный) и чтение LMS отложены на этап 4** — сознательно, так как относятся к сборке логики, а не к текстам промптов.

## Что унаследует этап 4

1. Подставлять `student_level` в системный промпт перед вызовом LLM.
2. Реализовать классификацию вопроса учебный/организационный и ветку чтения `knowledge_base/lms_stub.json` в обход векторного поиска.

Проверено реальными вызовами `ask_curator()`: на вопросе про непонимание циклов модель дала конкретный совет (пересмотреть раздел лекции, попробовать пример) со ссылкой на источник; на вопросе «что делать дальше после модуля 4» — рекомендации опирались только на структуру курса (следующий модуль, темы итогового проекта из FAQ), без выдуманных внешних ресурсов. `run_tests.py` — без регрессий.

## Обновление (2026-07-09): `few_shot.txt` стал реальным few-shot, а не только справочником

Изначально (см. «Важно» в разделе выше) было решено, что `few_shot.txt` — это справочный материал для человека при написании промпта, а не диалоговая история для передачи в API. Пользователь напомнил, что исходная цель формулировалась как «few-shot обучение» — то есть примеры должны реально идти в контекст модели, а не только жить в документации. Это решение было принято мной единолично на этапе 3 и не выносилось на явное обсуждение — по факту это был пробел, а не осознанный компромисс.

**Решение:** передавать в каждый вызов `ask_curator()` не все 7 пар из `few_shot.txt`, а только 3 отобранных примера — как реальные `user`/`assistant` сообщения перед финальным вопросом:

1. **Обычный хороший ответ** (словарь `.get()`) — эталон формата и ссылки на источник; без него модель видела бы в контексте только отказы.
2. **Честный отказ (Django)** — целится в найденную ранее слабость (см. `docs/stage4.md`): модель иногда сначала писала «не нашёл ответа», но затем всё равно поясняла из общих знаний.
3. **Отказ вне компетенции (смена оценки)** — другой тип отказа, демонстрирует стойкость под эмоциональным давлением студента («учитывая твои старания...» в исходном примере).

Остальные 4 примера (продление дедлайна, LMS-ответ, сложность материала, рекомендации) не включены — они близки по типу поведения к уже выбранным, а раздувать каждый вызов лишними токенами ради дублирующего сигнала не оправдано.

**Реализация (`curator.py`):** контекст в каждом few-shot примере — реальный чанк, полученный через `search_kb()` (не выдуман), чтобы формат совпадал с тем, что модель видит в проде. Из `few_shot.txt` берётся только пара вопрос/хороший-ответ — «плохой ответ» и разбор «почему плохо» не передаются модели, это комментарий для человека. `build_few_shot_messages()` разворачивает примеры в чередующиеся `user`/`assistant` сообщения, вставляются между системным промптом и финальным вопросом.

**Результат теста:** повторный прогон ранее нестабильного случая («Чем отличается список от кортежа?», `student_level="начинающий"`) — 3 из 3 повторов честно останавливаются на отказе, без утечки объяснения из общих знаний (раньше утечка воспроизводилась не всегда, но регулярно). Остальные сценарии (учебный вопрос, LMS, отказ по оценке, отказ по дедлайну) — без регрессий. `run_tests.py` — hit@4 = 15/15 = 100%, без изменений (few-shot не влияет на retrieval, только на финальную генерацию ответа).

## Обновление (2026-07-09): тексты 3 отобранных примеров вынесены в `prompts/few_shot_examples.json`

Первая реализация выше держала `FEW_SHOT_EXAMPLES` прямо в `curator.py` как питоновский литерал — пользователь справедливо указал, что это противоречит принципу, зафиксированному в этом же файле («тексты промптов — отдельные файлы в `prompts/`, не зашиты в код Python»). Вынесено в `prompts/few_shot_examples.json` — простой JSON-массив из 3 объектов `{context, question, answer}`, читается через `json.loads()` в две строчки, без самодельного парсинга (в отличие от текстовой разметки `few_shot.txt` с `=== Пример N ===`). В `curator.py` остался только комментарий с обоснованием, какие 3 примера и почему выбраны — сами тексты полностью переехали в `prompts/`.

Проверено: `run_tests.py` без регрессий (100%), проблемный кейс со «список vs кортеж» по-прежнему стабильно честный (2 из 2 повторных прогонов после рефакторинга) — перенос данных в JSON не изменил поведение.

## Обновление (2026-07-09): `few_shot_examples.json` → `few_shot_examples.txt`

Пользователь заметил, что JSON выбивается из общего стиля папки `prompts/` — `system_prompt.txt` и `few_shot.txt` оба текстовые, а этот файл единственный и по формату, и по расширению. JSON изначально выбран ради простоты чтения (`json.loads()` без самодельного парсера) — но однородность файлов в `prompts/` оказалась важнее.

**Решение:** `few_shot_examples.json` заменён на `few_shot_examples.txt` — та же разметка `=== Пример N ===`, что уже используется в `few_shot.txt`, с разделами `КОНТЕКСТ:`/`ВОПРОС:`/`ОТВЕТ:` внутри каждого примера. В `curator.py` вместо `json.loads()` — `load_few_shot_examples()`: `re.split()` по маркерам примеров + `re.match()` по маркерам разделов внутри блока, с явной ошибкой (`ValueError`), если какой-то пример не разобрался.

Проверено: сгенерированный `.txt` побайтово сравнён с исходным JSON (все 3 поля всех 3 примеров совпали посимвольно), `run_tests.py` — без регрессий, проблемный кейс со «список vs кортеж» — по-прежнему стабильно честный. Теперь все файлы в `prompts/` единообразны (обычный текст).

## Обновление (2026-07-10): промпты читаются с диска один раз за процесс

Пользователь спросил, срабатывает ли `load_few_shot_examples()` при каждом вызове `ask_curator()` или только один раз. Ответ: один раз — `FEW_SHOT_EXAMPLES = load_few_shot_examples()` и `SYSTEM_PROMPT_TEMPLATE = (...).read_text(...)` оба выполняются на уровне модуля, то есть при первом `import curator` в процессе; Python кэширует модуль в `sys.modules`, повторные импорты код заново не выполняют. Это не новое поведение, добавленное парсером, — так `system_prompt.txt` читался с самого начала (этап 3).

**Практическое следствие (важно задокументировать, легко забыть):** правка любого файла в `prompts/` не подхватится в уже запущенном процессе — например, в ноутбуке `demo.ipynb` с активным кернелом. Нужен перезапуск процесса (`Kernel → Restart` в Jupyter, либо просто новый запуск скрипта). Это тот же класс проблемы, что уже описан в `docs/stage4.md` про `NameError` при пропущенном импорте в свежем кернеле, но с другой причиной — там не выполнена ячейка, здесь процесс просто не видит изменений на диске, потому что данные уже закэшированы в памяти.

Добавлено в README.md (раздел «Промпты и их логика») и в Documentation.docx (раздел 3.2) как явная оговорка.
