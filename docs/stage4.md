# Этап 4 — Сборка и интерфейс

## Что сделано

- `curator.py` — главная функция `ask_curator(question, student_level="начинающий")`. Пайплайн одного вопроса:
  1. Классификация вопроса через отдельный (короткий) вызов LLM — `"lms"` (расписание/дедлайны) или `"kb"` (всё остальное, включая правила ДЗ и оценки).
  2. Сборка КОНТЕКСТа под найденный тип: либо полный текст найденных чанков из `search_kb()` (этап 2), либо всё содержимое `knowledge_base/lms_stub.json` целиком.
  3. Подстановка `student_level` в системный промпт (этап 3) через `.format()`.
  4. Вызов `ChatOpenAI` (`gpt-4o-mini`) с системным промптом + КОНТЕКСТом + вопросом.
- `demo.ipynb` — ноутбук с примерами диалогов, покрывающими все ветки: учебный вопрос, организационный (LMS), запрещённые темы (оценка, перенос дедлайна), честный отказ (Django-кейс из этапа 2), два примера адаптации под `student_level`.
- В `requirements.txt` добавлены `jupyter` и `ipykernel` (нужны для интерфейса-ноутбука, отсутствовали с этапа 2).



## Принятые решения и почему

- **Классификация вопроса — отдельный вызов LLM, а не ключевые слова.** Вопросы про сроки перефразируются слишком по-разному («когда дедлайн», «до какого числа сдавать», «во сколько вебинар по функциям»), чтобы надёжно ловить их регуляркой. Модель классификации — та же `gpt-4o-mini`, что и для финального ответа: два вызова LLM на вопрос, но модель дешёвая, для учебного объёма это не проблема.
- **LMS-контекст — весь** `lms_stub.json` **целиком, без парсинга/фильтрации по конкретному заданию.** Файл маленький (расписание + 6 заданий), проще отдать его целиком в контекст LLM, чем городить отдельный поиск/индекс по JSON.
- `build_kb_context()` **не переиспользует** `format_results()` **из** `search_kb.py` — та функция обрезает текст чанка до 180 символов (нужна для отладочного вывода в консоли), а для реального контекста LLM нужен текст чанка целиком. Написана отдельная функция с полным текстом и той же разметкой источника, что ожидает системный промпт (`[тип, модуль N, тема] (источник: имя_файла)`).
- **Название модели чата вынесено в** `.env` **(**`CURATOR_CHAT_MODEL`**)**, а не зашито строкой в `curator.py` — `CHAT_MODEL = os.getenv("CURATOR_CHAT_MODEL", "gpt-4o-mini")`, с тем же значением по умолчанию, чтобы код продолжал работать и без переменной. Позже по той же схеме в `.env` вынесен и `EMBEDDING_MODEL` для `index_kb.py`/`search_kb.py` (см. обновление в `docs/stage2.md`) — со сменой значения обязательна полная переиндексация, это выполнено вручную.



## Важная находка: частичное нарушение честности при отказе

На пограничном вопросе «Чем отличается список от кортежа?» (тема упомянута в базе одной фразой, но не раскрыта — честный отказ здесь обоснован) обнаружено:

- на `student_level="начинающий"` модель сначала пишет «в материалах курса я не нашёл ответа», но **затем всё равно поясняет разницу списка и кортежа из общих знаний** — прямое нарушение раздела «Источники и честность» системного промпта;
- на `student_level="продвинутый"` (изначально называлось «уверенный», переименовано 2026-07-09 — см. `docs/stage3.md`) в том же прогоне модель останавливается на отказе, не поясняя.

Это ограничение самой LLM (`gpt-4o-mini` не всегда строго держит инструкцию до конца ответа), а не ошибка сборки `ask_curator()` — воспроизводимость зависит от конкретного прогона и уровня студента. Есть ощущение, что более длинный/подробный стиль ответа для «начинающего» повышает риск такой утечки: модели "есть что сказать", и это перевешивает более раннюю инструкцию промпта.

**Как учесть дальше:** отметить как известное ограничение в итоговой документации (этап 6); при логировании (этап 5) полезно будет фиксировать не только факт отказа/неотказа, но и не проверять ли пост-фактум содержимое ответа при пустом контексте (`context == ""`) — вариант для будущего усиления, не реализовано намеренно (усложнение сверх задачи этапа 4).

## Как запустить интерфейс (`demo.ipynb`)

```
cd D:\Cursor2\МодульФриланс2Урок13\course-curator

```

(вместо `notebook` можно `jupyter lab`, если привычнее интерфейс JupyterLab). Команда откроет браузер с ноутбуком.

Ноутбук уже исполнен реальными вызовами API — ячейки можно просто прочитать сверху вниз и увидеть готовые ответы куратора на все 7 примеров.

**Важно при открытии ноутбука заново:** сохранённый вывод ячеек виден сразу, но это не значит, что кернел «помнит» переменные — при свежем запуске Jupyter память кернела пуста, пока ячейки не выполнены заново в этой сессии. Если нажать `Shift+Enter` сразу на ячейке в середине ноутбука (например, на примере с адаптацией), получится `NameError: name 'ask_curator' is not defined` — это не баг кода, а следствие пропущенного импорта.

Правильный порядок:

1. Сначала выполнить самую первую код-ячейку (`from curator import ask_curator`) — без неё ничего не будет определено.
2. Либо проще: `Cell → Run All` (или `Kernel → Restart Kernel and Run All Cells`) — выполнит весь ноутбук по порядку от начала до конца.
3. После этого уже можно пересчитывать отдельные ячейки через `Shift+Enter` или добавлять новую код-ячейку в конец для своего вопроса:

```python
print(ask_curator("Свой вопрос сюда", student_level="начинающий"))
```

**Как добавить новую ячейку:** проще всего в самый конец ноутбука (после последнего примера с адаптацией под уровень), чтобы не сломать порядок существующих примеров.

1. Кликнуть на последнюю ячейку, чтобы она была выделена.
2. Нажать клавишу `B` (Below) — добавится новая пустая код-ячейка сразу под ней (`A` — добавить выше). Либо через меню `Insert → Insert Cell Below`, либо кнопкой `+` на панели инструментов.
3. Кликнуть в новую ячейку, вписать код и выполнить `Shift+Enter`.

Нюанс: `A`/`B` работают только в command mode (ячейка выделена синей рамкой слева, а не в режиме редактирования текста с зелёной рамкой). Если вместо добавления ячейки буква "b" печатается в код — нажать `Esc`, чтобы выйти в command mode.
